Fields

info: MinerInfo

Contains static info about this miner

pre_commit_deposit: TokenAmount

Total funds locked as pre_commit_deposit

locked_funds: TokenAmount

Total unvested funds locked as pledge collateral

pre_committed_sectors: Cid

Sectors that have been pre-committed but not yet proven. Map, HAMT<SectorNumber, SectorPreCommitOnChainInfo>

sectors: Cid

Sectors this miner has committed Array, AMT

proving_period_start: ChainEpoch

The first epoch in this miner's current proving period. This is the first epoch in which a PoSt for a partition at the miner's first deadline may arrive. Alternatively, it is after the last epoch at which a PoSt for the previous window is valid. Always greater than zero, his may be greater than the current epoch for genesis miners in the first WPoStProvingPeriod epochs of the chain; the epochs before the first proving period starts are exempt from Window PoSt requirements. Updated at the end of every period by a power actor cron event.

new_sectors: BitField

Sector numbers prove-committed since period start, to be added to Deadlines at next proving period boundary.

sector_expirations: Cid

Sector numbers indexed by expiry epoch (which are on proving period boundaries). Invariant: Keys(Sectors) == union(SectorExpirations.Values()) Array, AMT[ChainEpoch]Bitfield

deadlines: Cid

The sector numbers due for PoSt at each deadline in the current proving period, frozen at period start. New sectors are added and expired ones removed at proving period boundary. Faults are not subtracted from this in state, but on the fly.

faults: BitField

All currently known faulty sectors, mutated eagerly. These sectors are exempt from inclusion in PoSt.

fault_epochs: Cid

Faulty sector numbers indexed by the start epoch of the proving period in which detected. Used to track fault durations for eventual sector termination. At most 14 entries, b/c sectors faulty longer expire. Invariant: Faults == union(FaultEpochs.Values()) AMT[ChainEpoch]Bitfield

recoveries: BitField

Faulty sectors that will recover when next included in a valid PoSt. Invariant: Recoveries ⊆ Faults.

post_submissions: BitField

Records successful PoSt submission in the current proving period by partition number. The presence of a partition number indicates on-time PoSt received.

next_deadline_to_process_faults: usize

The index of the next deadline for which faults should been detected and processed (after it's closed). The proving period cron handler will always reset this to 0, for the subsequent period. Eager fault detection processing on fault/recovery declarations or PoSt may set a smaller number, indicating partial progress, from which subsequent processing should continue. In the range [0, WPoStProvingPeriodDeadlines).
